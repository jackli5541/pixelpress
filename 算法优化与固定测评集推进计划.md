# 算法优化与固定测评集推进计划

## 1. 目标与范围

在不破坏既有相册主流程的前提下，逐步优化清洗、章节聚类和页面规划算法，并建立可复现、可比较、可回退的固定测评机制。

本计划只覆盖算法输入、离线评测、灰度发布和回归验证。全链路指标口径沿用《AI相册书算法评估指标设计.md》，上线风险与产品能力边界参考《AI相册书项目上线审计与实施计划.md》。

## 2. 当前基础与约束

### 已具备的基础

- 业务流程已拆分为 `upload -> cleaning -> chapters -> planning -> export`，任务由 ARQ 异步执行。
- 任务按内容版本生成幂等键，并会校验请求版本，能够避免旧任务覆盖新结果。
- 清洗、聚类和规划都保留规则回退；算法或 Provider 失败不应阻断用户继续完成相册。
- 渲染产物按 `content_revision` 和 `render_revision` 管理，内容改变后会自动失效并重建。

### 必须遵守的原则

1. 每次算法、规则、Prompt 或模型版本变更，都必须能回放到同一输入数据集。
2. 任何智能决策都必须保留可解释字段、版本号、置信度和规则回退原因。
3. LLM 可以参与语义补充和候选排序，但不能作为唯一的结构正确性保障。
4. 先优化上游特征，再优化下游生成；没有可靠图像和元数据特征时，章节与排版无法稳定提升。

## 3. 固定测评集建设

### 3.1 候选素材分组

| 数据集 | 当前候选 | 用途 | 不适合作为 |
| --- | --- | --- | --- |
| `album-v1` | 根目录 `测试例子/` 的相机照片 | 章节聚类、页面规划、端到端渲染 | 清洗鲁棒性的唯一基准 |
| `robustness-v1` | 根目录 `测试例子2/` 的截图、评论图、分享图和混合图片 | 清洗过滤、低质量与非相机图片处理 | 典型家庭相册的质量基准 |

现有素材只能作为候选输入。两个目录目前被 `.gitignore` 忽略，不能依赖开发机本地文件作为 CI 或回归测试的数据来源。

### 3.2 数据资产规范

为每个数据集建立版本化的 manifest，例如 `evaluation/manifests/album-v1.json`。manifest 必须进入 Git，原图应放在受权限控制的对象存储或 Git LFS 中。

每张图片至少记录：

- 稳定样本 ID、SHA-256、原始文件名、MIME 类型、宽高和可用 EXIF 元数据。
- 数据集版本、来源授权状态、是否包含人像或敏感内容。
- 清洗真值：保留/删除、原因标签、重复组 ID、组内首选图。
- 聚类真值：事件 ID、章节边界、允许的歧义边界。
- 规划约束：必须入页的照片、禁止同页的照片、允许模板范围和封面图。

原图内容或 manifest 任一变化，都必须创建新版本，禁止覆写既有 `v1`。

### 3.3 标注与复核

1. 先由两名标注人独立完成清洗和章节标注。
2. 对不一致项记录争议原因，并由产品负责人确定最终真值或“允许多解”规则。
3. 页面规划不对标题和文案做逐字真值；只标注可验证的结构约束。
4. 每次评测报告都要记录数据集版本、算法版本、模型版本和配置哈希。

## 4. 算法推进顺序

### 阶段 A：特征与评测底座

目标：让算法改动可以稳定回放和比较。

- 建立 manifest、样本下载校验、哈希校验和本地/CI 注入方式。
- 上传时提取并落库 EXIF 时间、时区、GPS、设备型号、尺寸和旋转信息。
- 增加评测运行器，输出 JSON 报告和可读摘要。
- 固化当前规则实现为 `baseline`，先得到现有质量基线。

验收：同一 manifest 在不同机器上能得到相同输入；缺少素材或哈希不一致时评测立即失败；能比较候选版本与 baseline 的差异。

### 阶段 B：照片清洗

目标：优先降低误删、重复误判和人工筛选成本。

- 用感知哈希替代仅按文件大小的重复分组，并保留近似重复的阈值与可解释距离。
- 增加清晰度、曝光、分辨率、横竖构图和连拍相似度特征。
- 在重复组中给出首选图评分，而不是只输出重复组。
- 保留规则回退和人工确认；删除建议必须可恢复。

核心离线指标：重复检测 precision/recall、删除建议 precision、误删率、重复组首选准确率。

发布门槛：候选版本不得降低高质量照片保留率；误删率超过既定阈值时禁止灰度。

### 阶段 C：章节聚类

目标：从单一时间切片升级为可解释的事件聚类。

- 先使用 `taken_at`、GPS、相机时间间隔和清洗结果建立确定性特征层。
- 再引入视觉 embedding、场景标签或人物聚合特征，形成候选事件簇。
- 使用 LLM 为候选章节命名、补充摘要或在候选分组中做受约束排序。
- 无 EXIF、模型失败或低置信度时保留时间聚类/单章回退。

核心离线指标：章节纯度、章节完整度、边界准确率、人工拆章/合章次数。

发布门槛：候选版本不得增加跨事件混章；低置信度相册必须显式标记并允许快速人工修正。

### 阶段 D：页面规划与排版

目标：先保证结构正确和视觉可用，再提升叙事性。

- 建立图片优先级、封面候选、横竖构图、重复度、质量和多样性评分。
- 使用约束求解或规则产生可行模板候选；模板容量、图片全覆盖、顺序和禁配约束必须由确定性逻辑保障。
- LLM 只在可行候选中提供排序、标题、副标题、图注和叙事建议。
- 对渲染结果执行图片低清、裁切风险、文本溢出和视觉回归检查。

核心离线指标：入页覆盖率、重复入页率、模板适配率、人工换图/换模板率、视觉回归通过率。

发布门槛：不得出现丢图、重复图、模板槽位不匹配、预览与导出不一致或渲染失败率上升。

### 阶段 E：灰度与持续迭代

目标：将离线质量提升转换为真实产品质量提升。

- 每次变更先跑固定集，对比 baseline 和当前线上稳定版本。
- 仅当 P0 指标不退化且目标指标提升时，进入小流量灰度。
- 灰度收集清洗改判、拆章/合章、换图/换模板、重渲染和导出失败原因。
- 指标退化或错误率超过阈值时，通过版本配置立即回退到规则实现或上一稳定版本。

## 5. 版本与报告规范

每次评测至少输出以下字段：

```json
{
  "dataset_version": "album-v1",
  "algorithm_version": "cleaning-phash-v1",
  "model_version": "optional-provider-model-id",
  "config_hash": "sha256",
  "run_at": "ISO-8601",
  "metrics": {},
  "regressions": [],
  "fallback_rate": 0
}
```

评测报告应保留在 CI 构件或受控对象存储中。Git 仅保存报告摘要、阈值配置和基准结论，避免提交含隐私的原始图片或完整推理结果。

## 7. 完成判定

本计划完成不以“接入某个模型”为标准，而以以下条件为准：

1. 算法版本可在固定输入上复现并与 baseline 对比。
2. 清洗、聚类、规划均具备真值或约束驱动的离线报告。
3. 任一版本均能明确其回退路径和适用范围。
4. 灰度放量有明确质量门槛，出现回归时可快速回退。
