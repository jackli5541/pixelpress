# 节点三开发文件

> 主题：分页规划节点（`pagination_planning_node`）最终实现方案  
> 位置：`backend/src/pixelpress_backend/graph/pagination_planning_node.py`  
> 当前状态：MVP 已落地，文档用于持续迭代更新

## 1. 节点定位

节点三负责把：

- `cleaned_photo_set`
- `chapter_plan`
- `constraints`

转换成：

- `page_plan`

它做的是整本书的“编辑级结构决策”，不是几何排版。

这个节点需要回答的问题是：

1. 这本书一共多少页
2. 每个章节分几页
3. 每一页是什么角色
4. 每一页候选放哪些照片
5. 这一页应该交给哪一类版式去渲染

## 2. 该节点最终要达到的效果

最终希望节点三输出的结果具备以下特征：

- 页数预算合理，不会因为照片多就机械堆叠
- 每章至少有起势页，章节结构清晰
- 高权重照片优先成为章节首页或重点页
- 普通照片更多进入拼图页或收束页
- `must_include` 一定进入书中
- `must_exclude` 一定不进入书中
- 页与页之间有节奏，而不是平均切片
- 输出结果可解释、可测试、可继续优化

## 3. 当前已实现的版本

当前代码已具备以下能力：

1. 对章节内照片做基础优先级排序
   - `must_include` 优先
   - `cover_photo_id` 可优先成为章节首页
   - `hero_person_id` 可提升主角人物照片排序
   - `keep` 优先于 `deprioritize`
   - `rank_weight` 高的优先
   - 重复照片在重点排序中降权
2. 根据章节数和照片数估算总页数
3. 给每章至少分配 1 页
4. 按章节照片量比例分配剩余页数
5. 生成页面角色
   - `chapter_opening`
   - `hero`
   - `transition`
   - `collage`
   - `detail`
   - `summary`
   - `ending`
6. 为每页生成 `candidate_photo_ids`
7. 为每页生成 `layout_family`
8. 为章节首页生成 `text_need = chapter_title`
9. 为每章生成 `chapter_page_budgets`
10. 生成节点级 `planning_summary`
11. 在满足条件时为 `hero` 页输出跨页候选
12. 在 `avoid_spread = true` 时禁止跨页候选
13. `collage` 页按章节叙事顺序和场景相近度成组
14. 相邻页面优先避免复用相同照片
15. 多个 `collage` 页在版式族上做交替，避免连续同构
16. `ending` 页在照片充足时可输出 2 张收束图
17. 通过章节强度评分分配章节预算，而不是只按照片数机械分配
18. 通过全书级主角曝光目标，优先让含主角章节获得重点页
19. 为同一输入生成多个候选分页预算，并用规则分做小规模 rerank
20. 在 `metadata.planning_debug` 中输出候选方案数、命中方案、主角曝光目标和章节强度，便于调试和复盘
21. 已接入“章节故事策划器适配层”，支持由大模型判断章节故事线和每页候选照片
22. 大模型未配置或返回异常时，自动回退到规则规划，不中断节点三输出
23. 在 `metadata.story_planner` 中输出章节主题、故事弧线和每页建议，便于调试与人工审阅

这是一版“规则规划器 + 可选 LLM 章节策划”的混合实现，已进入“第四阶段：规则增强 + 小规模 rerank + 章节故事策划器”。

## 4. 最终实现建议分三层

### 4.1 第一层：章节内照片排序

目标：先得到章节内“推荐出场顺序”。

排序优先级建议如下：

1. `must_include`
2. 主角人物照片（后续引入 `hero_person_id`）
3. `keep`
4. `rank_weight`
5. 时间顺序

最终应形成：

```python
ordered_photo_ids_for_chapter: list[str]
```

### 4.2 第二层：页面角色规划

目标：先决定每一页是什么页，再决定放哪些照片。

推荐页面角色：

- `chapter_opening`：章节起势页
- `hero`：主视觉页
- `collage`：多图叙事页
- `ending`：章节收束页

后续可扩展：

- `transition`
- `appendix`
- `spread_hero`

### 4.3 第三层：页级照片分配

目标：按照页面角色，从章节有序照片池中挑选该页候选照片。

建议规则：

- `chapter_opening`：1 张强图
- `hero`：1 张强图
- `collage`：2 到 4 张照片
- `ending`：1 到 2 张照片

### 4.4 第四层：章节故事策划器

目标：在不破坏现有契约的前提下，让大模型判断“这一章可以讲成什么故事”，并给出每页照片建议。

当前实现策略：

1. 先由规则层算出章节页数预算和基础页面角色
2. 再将章节照片摘要、约束、页数预算交给章节故事策划器
3. 若配置了大模型：
   - 调用 OpenAI 兼容接口
   - 返回结构化 JSON
4. 若未配置或调用失败：
   - 自动回退到启发式故事策划器
5. 规则层对模型输出做校验：
   - 非法页角色忽略
   - 非本章照片忽略
   - 不足的候选照片用规则层补齐
6. 最终仍由节点三输出结构化 `page_plan`

当前返回结构：

```json
{
  "chapter_id": "chapter-001",
  "provider": "openai_compatible | heuristic_fallback | llm-test",
  "chapter_theme": "string",
  "story_arc": "string",
  "page_suggestions": [
    {
      "page_index": 0,
      "page_role": "chapter_opening",
      "candidate_photo_ids": ["p1"],
      "narrative_purpose": "string",
      "reason": "string"
    }
  ]
}
```

## 5. 最终希望输出成什么样

示例输入：

- 章节 A：`[p1, p2, p3, p4, p5]`
- 章节 B：`[p6, p7, p8]`
- `must_include = [p4]`
- `must_exclude = [p7]`

一个理想的 `page_plan` 效果应该类似：

| 页码 | 章节 | 页面角色 | 候选照片 | 版式族 | 说明 |
| --- | --- | --- | --- | --- | --- |
| 1 | A | chapter_opening | [p4] | chapter_cover | 强制包含图优先放章节首页 |
| 2 | A | hero | [p1] | single_full_bleed | 章节主视觉 |
| 3 | A | collage | [p2, p3] | grid_nine | 中段叙事 |
| 4 | A | ending | [p5] | double_side_by_side | 章节收束 |
| 5 | B | chapter_opening | [p6] | chapter_cover | 第二章开场 |
| 6 | B | hero | [p8] | single_full_bleed | 第二章重点页 |

这个结果说明：

- `p4` 被强制纳入且优先展示
- `p7` 被彻底排除
- 章节 A 因照片多而分到更多页
- 高优先级图进入首页和重点页

## 6. 每页有哪些照片的最终选择方案

这是节点三最关键的部分。

### 6.1 章节首页 `chapter_opening`

目标：

- 作为章节第一印象
- 需要一张代表性最强的图

选择规则建议：

1. 优先 `must_include`
2. 再优先包含 `hero_person_id` 的图
3. 再优先 `rank_weight` 最高图
4. 若有 `cover_photo_id`，可给额外加分

期望结果：

- 只放 1 张照片
- 适合 `chapter_cover`

### 6.2 主视觉页 `hero`

目标：

- 展示该章节中最值得单页展示的照片

选择规则建议：

1. 从未使用或未在重点页使用的照片中挑
2. 优先高 `rank_weight`
3. 优先主体明确、画面干净的照片
4. 未来可引入横竖比和分辨率信号

期望结果：

- 只放 1 张照片
- 适合 `single_full_bleed`

### 6.3 拼图页 `collage`

目标：

- 用多图表达叙事过程，而不是突出单张震撼感

选择规则建议：

1. 选同一章节中时间相邻的照片
2. 选视觉信息互补的照片
3. 避免 2 张几乎一样的连拍图
4. 中低优先级图优先进入拼图页

当前实现：

- 每页优先取 2 张
- 先取章节叙事顺序中尚未使用的 anchor 图
- 再从时间邻近、场景标签相近、人物重叠的照片里补齐
- 相邻页已使用的照片会被优先避开

最终优化版建议：

- 2 图页：偏关系对照
- 3 图页：偏小故事展开
- 4 图页：偏摘要页

### 6.4 收束页 `ending`

目标：

- 让章节在视觉上自然结束

选择规则建议：

1. 使用章节剩余较好的照片
2. 优先情绪收束图、细节图、补充图
3. 不强行再塞最强图

期望结果：

- 1 到 2 张照片
- 更适合 `double_side_by_side`

## 7. 页数分配的最终方案

### 7.1 总页数估算

建议先维持当前实现思路，再逐步增强：

```text
总页数 = 章节基础页数 + 内容页数
章节基础页数 = 每章至少 1 页
内容页数 = 按每 2 张照片约 1 页估算
最终结果受 min_pages / max_pages 约束
```

后续增强方向：

- 针对高质量强图提高单图页占比
- 针对大量弱图提高拼图页占比
- 针对不同 `scene_mode` 做不同密度策略

### 7.2 章节页数分配

当前建议保留：

1. 每章先给 1 页
2. 剩余页按章节照片数比例分配
3. 余数补给照片更多的章节

后续可增强为：

- 强章节多分 1 页
- 弱章节少分 1 页
- 有主角人物密集曝光的章节优先多给重点页

## 8. 页面角色生成的最终方案

当前实现的页面角色规则：

- `1 页`：`chapter_opening`
- `2 页`：`chapter_opening + hero`
- `3 页`：`chapter_opening + hero + ending`
- `4 页`：`chapter_opening + hero + collage + ending`
- `5 页`：`chapter_opening + hero + transition + summary + ending`
- `6 页`：`chapter_opening + hero + transition + collage + summary + ending`
- `7 页`：`chapter_opening + hero + transition + collage + detail + summary + ending`
- `8+ 页`：在 `transition` 之后交替插入 `collage / detail`，最后以 `summary + ending` 收束

最终增强方向：

- 大章节可以拥有多个 `hero`
- 存在跨页候选时插入 `spread_hero`
- 基于章节强度和场景变化密度动态调整 `transition / detail / summary`

## 10. 建议的优化路线

### 阶段 1：可运行规则版

目标：

- 先把章节分页、页面角色、候选照片分配跑通

验收标准：

- 输出 `PagePlan`
- 通过单元测试
- `must_include / must_exclude` 生效

### 阶段 2：更像“相册书”的规则版

目标：

- 引入主角人物优先
- 引入章节内节奏增强
- 引入拼图页分组优化

验收标准：

- 强图更稳定进入首页/重点页
- 拼图页不再只是顺序拿图

### 阶段 3：局部优化版

目标：

- 避免相邻页重复照片
- 避免连续多页同构
- 更好地平衡重点页和摘要页
- 引入章节强度评分和全书级主角曝光控制
- 生成多个候选分页预算并做 rerank

验收标准：

- 页面更有节奏
- 输出仍然稳定、可解释

## 11. 节点三不该做什么

必须严格保持边界：

- 不重新做照片清洗
- 不做裁切框求解
- 不做几何坐标生成
- 不修改 `AlbumState`
- 不修改 `TaskState`
- 不落库
- 不输出最终 `BookLayout`

节点三只负责：

- 规划“哪些页”
- 规划“每页什么角色”
- 规划“每页候选哪些照片”

## 12. 测试建议

节点三必须长期维持独立测试。

至少覆盖：

1. 多章节分页是否合理
2. `min_pages / max_pages` 是否生效
3. `must_exclude` 是否完全排除
4. `must_include` 是否优先进入首页或重点页
5. 高 `rank_weight` 是否优先成为首页候选
6. 照片不足时是否仍能生成非空页面候选
7. 空章节时是否行为可控
